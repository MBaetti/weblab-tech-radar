[[section-technical-risks]]
== Risiken und technische Schulden

=== Risiken

[cols="1,1,2" options="header"]
|===
| Risiko | Priorität | Massnahme

| Keine Authentifizierung
| Hoch
| Die Technologie-Radar Administration ist aktuell ohne Login zugänglich. Für einen produktiven Betrieb muss eine Authentifizierung z.B. mit Auth0 integriert werden.

| Auditing fehlt
| Mittel
| Es werden keine Änderungen an den Technologie-Einträgen protokolliert. Ein Log sollte Informationen zu Änderungszeitpunkt, Benutzer (im Zusammenhang mit Risiko1) und Art der Änderung aufzeichnen.

| Kein HTTPS
| Mittel
| Die Kommunikation zwischen Client und Server erfolgt zurzeit unverschlüsselt über HTTP. Für den produktiven Betrieb sollte HTTPS mit einem gültigen Zertifikat eingerichtet werden.

| Kein Backup-Konzept
| Mittel
| Die PostgreSQL-Daten liegen in einem Docker-Volume ohne automatisiertes Backup.
|===

=== Technische Schulden

[cols="1,1,2" options="header"]
|===
| Schuld | Priorität | Beschreibung

| Keine automatisch generierte API-Dokumentation
| Niedrig
| Die REST-API ist nicht über OpenAPI/Swagger dokumentiert. Dies wurde bewusst aufgrund mangelnder Zeit zurückgestellt.

| Keine CI/CD-Pipeline
| Mittel
| Build, Tests und Deployment werden manuell ausgeführt. Eine automatisierte Pipeline würde dies vereinfachen.
|===
