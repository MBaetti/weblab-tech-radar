== Querschnittliche Konzepte

=== Persistierung

Das Datenmodell wird zentral im Prisma Schema definiert.
Prisma generiert daraus einen typensicheren Client, der im Server für alle Datenbankzugriffe verwendet wird.
Es können Initialisierungsdaten über ein Seed-File definiert und eingespielt werden.
Die Daten werden in einem Docker Volume persistiert.

=== API Kommunikation

Client und Server kommunizieren ausschliesslich über eine REST-API unter dem Pfad `/api/technologies`.
Im Produktionsbetrieb/Compose-Stack leitet Nginx alle Anfragen an `/api/` als Reverse Proxy an den Server weiter.
In der lokalen Entwicklung übernimmt ein Angular Proxy dieselbe Aufgabe.

Der Client kapselt sämtliche HTTP-Aufrufe in einem zentralen TechApiService.
Dieser verwaltet den Zustand der Technologieliste als Angular Signal und aktualisiert ihn nach jeder erfolgreichen Mutation.

=== Validierung

Die Eingabevalidierung erfolgt auf zwei Ebenen:

**Client**: Angular Reactive Forms mit Validatoren verhindern das Absenden unvollständiger Formulare.

**Server**: Der Controller auf dem Server prüft jeden Request auf gültige Typen, Pflichtfelder und Duplikate. Bei Fehlern werden entsprechende HTTP-Statuscodes zurückgegeben.

=== Fehlerbehandlung

Auf dem Server fängt eine zentrale Error-Handler-Middleware alle nicht abgefangenen Fehler ab und liefert eine einheitliche JSON-Antwort mit Statuscode und Fehlermeldung zurück.
